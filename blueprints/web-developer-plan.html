Web Developer — Complete System Plan

A publishing and page-building system for 7 websites. Takes content from the Content Studio and turns it into live pages.
Content Studio → Web Developer → Live Pages
← Blueprints / Web Developer

What This Is

The Web Developer is a publishing and page-building system for Dave Hajdu's websites. It takes content - from the Content Studio or from direct conversation - and turns it into live pages on any of Dave's 7 websites. Each site has a different tech stack, design system, and publishing process. The Web Developer knows how to build for all of them.

It's the second half of a two-part pipeline:

Content Studio (/studio)          Web Developer (/publish, /build)
────────────────────────          ──────────────────────────────
Produces the content:             Builds and deploys:
  blog.md                →         Blog post (HTML, JSX, JSON)
  social.md              →         (posted manually by Dave)
  seo.md                 →         Meta tags, schema, sitemap
  page.md                →         Web page (any layout)
  images.md / card.png   →         Optimized WebP images

Architecture

Three-Layer Pattern (Same as Content Studio)

Both systems share the same foundational architecture: data, logic, and workflow.

Layer Content Studio Web Developer
Data (context files) Voice, audience, channel rules per brand Tech stack, templates, components, design tokens per site
Logic (instructions) CLAUDE.md - workflow rules, editorial lenses CLAUDE.md - build rules, deploy rules, git discipline
Workflow (skills) /studio, /onboard /publish, /build

Skill 1: /publish (Blog Publishing)

Purpose

Takes an approved blog post from the Content Studio and publishes it to a website. Same steps every time. No creative decisions - it's a machine.

Workflow: 4 Phases, 9 Steps

Phase 1: Gather

1 Locate content - find blog.md + seo.md in content-studio/{project}/content/{slug}/

2 Read and parse - extract title, body, meta tags, schema, hero image

Phase 2: Build

3 Optimize image - convert to WebP (Pillow or sips), max 1200px, under 200KB

4 Build page - construct full page using site's template, component patterns, and design tokens. Includes:

  • Document head (meta, OG, Twitter Card, JSON-LD from seo.md)
  • Site navigation with correct relative paths
  • Hero/header with breadcrumb, category, title
  • Post meta (date, reading time, category)
  • Article content (markdown → site-specific HTML/JSX components)
  • Internal links (3-4) and external links (1-2) - 5 total per post
  • FAQ section (from seo.md structured data)
  • Related reads (2 recent posts from nav-index)
  • CTA section
  • Prev/next navigation
  • Footer
Phase 3: Index

5 Update blog listing - insert new post card at top of blog index

6 Update navigation chain - add post to nav-index.md, update prev/next links

7 Patch previous post - add "next" link pointing to new post

8 Update sitemap - add new URL to sitemap.xml

Phase 4: Ship

9 Git commit and deploy - stage explicit files, commit with blog: publish "{title}", run deploy command

Skill 2: /build (Page Creation) — Planned

Purpose

Create any page on any website. Landing pages, product pages, info pages, form pages. More flexible than /publish - every page is different, requires conversation and creative decisions.

Key Difference

/publish is a machine: same inputs, same steps, every time.

/build is creative: different layouts, different components, needs more conversation.

Content Sources

  1. From the Content Studio - if a page.md exists in the studio, read it
  2. Direct conversation - Dave describes what the page should contain, /build creates it
  3. Hybrid - Dave gives the idea, /build develops the content and builds it in one pass

Workflow: 5 Phases, 14 Steps

Phase 1: Scope

1 Which site? - read site-config.md

2 What page? - landing page, info page, product page, form page, custom

3 Content source? - content studio, direct input, or hybrid

Phase 2: Content

4 Gather or develop content - if from studio, read page.md. If direct, ask Dave what goes on the page - sections, messaging, CTAs, images

5 Confirm structure - present a section-by-section outline before building

Phase 3: Build

6 Select layout - read site's page-patterns.md. Pick the right layout pattern (hero + sections, sidebar, full-width, etc.)

7 Build page - construct using site's design system, components, and conventions

8 Forms - if the page has a form, read site's forms.md for backend, spam protection, validation, confirmation UX

9 SEO - meta tags, OG, structured data

Phase 4: Review

10 Present for review - show the page to Dave, iterate

11 Approval gate - explicit confirmation before deploying

Phase 5: Ship

12 Place file - save to correct location per site architecture

13 Update navigation - add to site nav if it's a core page

14 Git commit and deploy

Per-Site Context Needed for /build

Each site needs these additional context files:

File What It Holds Why
page-patterns.md Layout patterns available on this site (hero + sections, sidebar, full-width, card grid, etc.) Different sites have different layouts
forms.md Form backends, spam protection, validation, confirmation UX Forms have specific security and UX concerns
component-library.md All reusable components (heroes, CTAs, cards, stats, testimonials, etc.) Build pages from existing pieces

Form Handling with Spam Protection

Every form built by /build includes spam protection by default

Static HTML Sites (CAIO Coach, Edge8)
  • Honeypot field - hidden input, bots fill it, humans don't
  • Time-based check - reject submissions faster than 3 seconds
  • Server-side validation if endpoint available
Next.js Sites (AIO, Dave Hajdu, Mahjong Tarot)
  • Honeypot field
  • Server action or API route for processing
  • Rate limiting middleware
  • Turnstile/reCAPTCHA if high-traffic form

Example: /build in Action

"3 Core Skills of an AI Officer" Page on AIO Website

Here's how Dave would use /build for this:

Dave: /build aio-website  -  I want a page on the three core skills of an AI officer

 1. Reads aio-website/context/site-config.md (Next.js 16, static HTML in /public/)
 2. Reads aio-website/context/page-patterns.md (available layouts)
 3. Asks Dave: "What are the three skills? What's the CTA? Is this a landing page or an info page?"
 4. Dave describes the content

    - Hero: "The Three Core Skills of an AI Officer"
    - Skill 1 section (card with description)
    - Skill 2 section
    - Skill 3 section

 6. Dave confirms

Content Studio Updates Needed

The Content Studio currently supports two content types: blog and social. To support /build, it needs a third: page content.

Current Content Types

Type Entry Output
Blog Idea → develop → style → outline → Shipper lens → draft → iterate → approve → SEO → deliverables → images blog.md, seo.md, social.md, images.md
Social Idea → channels → draft → iterate → save social.md

New Content Type: Page

Type Entry Output
Page Idea → define sections → outline → Shipper lens → draft → iterate → approve → SEO page.md, seo.md

Page Workflow

  1. Intake - what page is this? What's the goal?
  2. Section planning - what sections does the page need? (hero, features, testimonials, CTA, FAQ, etc.)
  3. Outline - section-by-section structure with key messaging per section
  4. Shipper lens - does every section earn its place? Is the page doing too much?
  5. Draft - write page.md with all section content
  6. Iterate - revise until approved
  7. SEO - meta, OG, structured data
  8. Handoff - content is ready for /build to turn into a live page
Note

Not all pages need the content studio. Simple structural pages (contact, pricing calculator) can go straight to /build. The studio is for content-heavy pages where the writing matters.
